## 2. Security & Data Handling

### Data Access Principles

| Principle | Statement |
|-----------|-----------|
| Minimal access | We only access data needed for the specific project |
| Client control | Access uses credentials you control; revocable at any time |
| Need-to-know | Only the project team (typically David McCann) accesses your data |
| No subcontracting | No offshore teams or subcontractors without explicit approval |
| Documentation | We document what data we accessed and when |

### Data Storage & Transfer

| Aspect | Standard |
|--------|----------|
| Data at rest | Stays on client systems where possible |
| Working copies | Encrypted storage when extracts needed |
| Data in transit | HTTPS/TLS encryption always |
| Email | No sensitive data via email attachments |
| Post-project | Working copies deleted after project completion |

### Security Measures

| Measure | Implementation |
|---------|----------------|
| Encryption | AES-256 for stored data; TLS 1.2+ for transit |
| Authentication | Strong passwords; 2FA where supported |
| Access logging | Documented access records |
| Secure development | No credentials in code; environment variables |
| Backup | Client data not backed up on our systems (stays with client) |
